> **本文件是平台实现 slice 的标准模板。**
> 复制本文件到 `slices/{product}/{platform}/{ability}.md`，按 `<!-- 指引: ... -->` 批注填写内容，填完后删除所有批注。
>

```yaml
---
id: {product}/{ability}       # 与 index.yaml 中的 id 一致
platform: {platform}          # ios / android / web / flutter / electron
api_docs:                      # 该平台对应的 API 参考文档链接
  - title: {API 类名/模块名}
    url: https://...
---
```

<!-- 指引: Frontmatter 字段说明：
     - id: 与 index.yaml 中的 slice id 完全一致
     - platform: 当前平台标识
     - api_docs: 该功能在该平台的 API 参考文档链接（接口签名、参数类型、返回值等）。
       与 slice 内容互补：slice 负责实现指南（代码示例、注意事项），api_docs 负责精确的接口参考。
       — 填写精确到类/模块级别的 URL（如 CoGuestStore 的 API 页面），不要填整个 SDK 首页
       — 可以有多条（如一个 slice 涉及多个 Store 的 API）

     其余元数据（name、tags、platforms、related）在产品级概览中维护，此处不重复。
-->

## 代码示例

<!-- 指引: 本 section 是平台 slice 的核心交付物。提供可直接运行的完整示例代码。
     定位是「零件」— 单个功能的完整实现。多个零件如何组装成完整场景，由 scenario 的平台实现文件负责。

     要求：
     1. 代码必须能编译能运行，不是伪代码
        — 包含必要的 import 语句
        — 包含完整的类/函数结构，不要只贴片段
     2. 按用户操作流程组织代码，用 MARK / region / 注释分隔各步骤
        — 例如：初始化 → 核心操作 → 事件监听 → 错误处理 → 清理
     3. 关键调用处用行内注释标注：
        — ⚠️ 易错点（如参数类型陷阱、命名容易混淆的地方）
        — 为什么这样写（不明显的设计决策）
     4. 每个失败分支必须有错误处理，不能只 print
     5. 如果该功能有多个角色（如主播端/观众端），分开写独立的示例
     6. API 签名细节不用单独列 section，通过代码中的调用自然体现即可
        — API 文档链接见 llms.txt 体系

     可组合性要求（为 scenario 组装做准备）：
     7. 对外暴露清晰的输入/输出 — 函数参数和回调/返回值要明确
     8. 不要硬编码其他 slice 的调用（如初始化登录），用注释标注依赖：
        「// 前置：登录完成（→ live/login-auth）」
     9. 状态变更要通过可观察的属性暴露（如 @Published / LiveData / EventEmitter），方便其他模块监听

     ❌ 不要写伪代码或用省略号 (...) 跳过逻辑
     ❌ 不要混入其他 slice 的职责代码
     ❌ 不要把多个功能耦合在一个类里
     ❌ 不要省略错误处理 — 每个失败分支都必须有处理

     质量标准：
     — 单独看：复制到项目补上业务参数即可跑通
     — 组合看：AI 读完多个 slice 的代码后，能理解各自的输入/输出边界，在 scenario 层完成组装
-->

## 平台特有注意事项

<!-- 指引: 本 section 是平台 slice 区别于产品级概览的核心价值所在。
     只写该平台独有的坑，产品级概览的通用最佳实践不要重复。

     每条注意事项的格式：
     ### {编号}. {一句话标题}
     — 先说现象/问题（开发者会遇到什么）
     — 再说原因（为什么会这样）
     — 最后给代码对比（✅ 正确写法 vs ❌ 错误写法）

     什么内容该写在这里：
     ✅ 该平台的类型陷阱（如 iOS 的 Int32 vs Int、Android 的 Int vs Long）
     ✅ 该平台的生命周期问题（如 Android Activity 重建、iOS 后台挂起）
     ✅ 该平台的内存/线程问题（如 iOS [weak self]、Android 主线程更新 UI）
     ✅ 该平台的权限行为差异（如 iOS 崩溃 vs Android 返回错误码）
     ✅ 该平台特有的错误码或异常行为

     什么内容不该写：
     ❌ 跨平台通用的业务逻辑（如「先登录再操作」→ 产品级概览已覆盖）
     ❌ API 用法说明（→ 代码示例已覆盖）
     ❌ 通用排障流程（→ 产品级概览已覆盖）

     质量标准：每条都应该是「不写出来，研发大概率会踩坑」的内容。
     如果一条注意事项对于有经验的该平台开发者来说是常识，就不用写。
-->

## 代码生成约束

<!-- 指引: 本 section 供 AI 在生成/验证代码时使用，是给机器读的硬性规则。
     与「平台特有注意事项」互补：
     — 注意事项 = 给人读的经验提醒
     — 代码生成约束 = 给 AI 读的可检查规则

     所有规则必须基于实际 SDK 行为，不允许凭经验推测。
-->

### 编译必要条件

<!-- 指引: 本 slice 代码能编译通过的最小条件。
     只写增量条件（通用条件见 login-auth slice）。
     — 必须导入的模块/包（精确到包名）
     — 本 slice 额外需要的 SDK 版本要求（如果高于基础要求）
     — 本 slice 额外需要的权限声明或配置

     如果没有增量条件，写「同 login-auth，无额外要求」
-->

### 生成规则

#### MUST（生成时必须包含）

<!-- 指引: 每条格式：编号 + 规则 + 违反时的具体后果
     例如：「1. **所有 Combine sink 必须 `[weak self]`** — 不用会导致循环引用，ViewModel 永远不释放」

     只写本 slice 独有的规则。跨 slice 通用规则（如 [weak self]、主线程更新）
     如果在多个 slice 中反复出现，考虑提到产品级概览或 base-setup 中。
     但如果该规则在本 slice 有特殊表现形式，仍需列出。
-->

#### MUST NOT（生成时绝不能出现）

<!-- 指引: 同上格式。重点写「看起来能跑但逻辑错误」的写法。
     例如：「不要在 applyForSeat 成功回调中开设备 — .success 仅表示申请发出，不是主播同意」
     这类问题编译器抓不到，只有了解业务语义才能避免。
-->

### 集成检查点

<!-- 指引: 假设目标是已有项目（不是从零开始的 demo），列出集成时需要确认的事项：
     — 是否与项目中已有的 SDK 初始化冲突
     — 是否需要合并到已有的生命周期方法中（而非新建）
     — 是否依赖其他 slice 的前置状态（引用具体 slice ID）
     — 对已有代码的侵入性（新增文件 vs 修改已有文件）
-->

### 可运行验证

     — 编译命令（该平台的构建命令）
     — 预期编译结果
     — 最小运行时验证（启动 App → 做什么操作 → 预期看到什么）
